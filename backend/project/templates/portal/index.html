Ethics & Compliance

Copy-ready boilerplate text for IRB / ethics board submissions.

Data Encryption at Rest

Use the following paragraph when describing how participant data is stored:

All behavioural data collected through CogFlow are encrypted at rest using AES-128 symmetric encryption (Fernet, RFC 7519 compliant) before being written to the database. Each result payload is independently encrypted with a server-managed key. Encryption occurs server-side immediately upon receipt; raw participant responses are never persisted in plaintext. Decryption requires authenticated researcher access and, where multi-factor authentication is enabled, a verified TOTP session within the preceding 24 hours.

Participant Identifier Pseudonymisation

Use the following paragraph when describing identifier handling:

Participant identifiers are pseudonymised prior to storage. External participant IDs (e.g., Prolific IDs, SONA codes) are hashed using a salted one-way function before being written to the database. The plaintext identifier is never stored. A researcher-visible truncated preview (first six characters of the hash) is retained solely for quality-control purposes and cannot be reversed to recover the original identifier.

Data Transmission Security

Use the following paragraph when describing data in transit:

All data transmitted between participants' browsers and the CogFlow platform server are encrypted in transit using TLS 1.2 or higher. The platform does not use persistent client-side storage (cookies carry only session tokens; no behavioural data are written to localStorage or IndexedDB). Session tokens expire on logout and are bound to the server-side session record.

Data Retention and Deletion

Use the following paragraph when describing retention policy:

Participant data are retained for the duration of the research project and for a post-completion period as specified in the approved ethics protocol, after which all result envelopes are permanently deleted from the database. Researchers may request deletion of data for individual participants or for an entire study at any time via the portal. All deletion events are recorded in an immutable audit log with timestamps and requesting-user identity.

Access Controls and Audit Logging

Use the following paragraph when describing access controls:

Access to participant data is restricted to the owning researcher account and platform administrators. All sensitive actions (data decryption, participant link generation, researcher assignment changes, data export, and deletion requests) are recorded in a tamper-evident audit log that captures the acting user, action type, target resource, timestamp, and outcome. Audit records cannot be modified or deleted by any user role.
